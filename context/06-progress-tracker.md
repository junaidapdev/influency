The status machine (`features/deals/status.ts`) is verified by typecheck + inspection (pending→in_progress on first posted; →posted when all posted) but the full create→mark-posted flow and the second-user RLS test need a real session.
- **jsonb (deliverables) + numeric (agreed_amount_sar) round-trip unverified at runtime.** Inserts pass the deliverables array directly (standard PostgREST behavior, no stringify) and the row schema coerces numeric → number; both are types-correct but not yet exercised against the live DB. Confirm on the first real deal write.
- **Payments direct-write lockdown DONE (chunk 04 review, migration `20260609160000`).** `UPDATE` on `payments` revoked from `authenticated`; `mark_payment_received` is now `SECURITY DEFINER` (locked `search_path`, schema-qualified, keeps explicit `user_id = auth.uid()` filters), so the RPC is the sole writer of a payment. Insert/update policies now also require the referenced `deal_id` to be owned by the caller (no cross-tenant FK).
- **Still open: `ad_deals` direct `status='paid'` write + `ad_deals.brand_id` cross-tenant FK (same classes as the payments fixes, in the merged chunk 03).** `ad_deals` can't simply revoke `UPDATE` (clients legitimately write `deliverables`/`status` for the toggle + cancel), so locking `'paid'` there needs a `SECURITY DEFINER` setter or a trigger; and the `ad_deals` insert policy doesn't yet require `brand_id` ownership (mirror of the payments `deal_id` fix). Both are within-tenant only (RLS isolates tenants). Worth a small follow-up migration.
- **Payments mark-received + atomicity unverified at runtime (chunk 04).** Atomicity is structural — the RPC is a single plpgsql function (one transaction), so a mid-function failure rolls back both writes — but the acceptance "force a failure and confirm rollback" and the all-paid→deal-`paid` flow need a real session + data. To run the atomicity test manually: temporarily add `RAISE EXCEPTION 'TEST'` after the payment UPDATE in the RPC, mark a payment received, confirm neither the payment nor the deal changed, then remove it.
- **`mark-received` saves the local calendar day (chunk 04 review fix).** `lib/date.ts#todayIsoDate()` builds `YYYY-MM-DD` from local fields, not `toISOString()` (UTC), so a payment marked after local midnight in a non-UTC zone records the correct day.
- **Dashboard numbers unverified against a hand calculation (chunk 05).** The `dashboard_summary` RPC is smoke-tested (valid JSON shape, runs clean) but the acceptance "every top-line number matches a hand calculation" + the second-user isolation test need a real session with seeded deals/payments. Month basis to confirm: invoiced + deal counts use deals' `created_at` month; collected uses payments' `received_date` month; `outstanding = invoiced − collected` (spec formula, so cross-month collections can make it negative — intended).
- **"Today" panel wired (chunk 06).** `useDashboard` now fetches real next-24h upcoming meetings + not-done reminders (due ≤ now+24h, so overdue ones show). The today queries are keyed by `today` so they refresh across day boundaries.
- **Meeting ↔ reminder orchestration is client-side, not atomic (chunk 06).** `useMeetings` creates the meeting then its reminder as two sequential SDK calls (reminders must be created in code, not a trigger — confirmed none added). If the reminder call fails after the meeting is created, you get a meeting without a reminder (recoverable by editing the meeting, which re-syncs). Reminders are non-financial, so this wasn't elevated to an edge-function RPC like payments; revisit if it proves flaky.
- **Meetings UI + reminder flow unverified at runtime (chunk 06).** The create→reminder-in-Today, edit-moves-reminder, cancel-clears-reminder, and payment "Send reminder" flows + the second-user RLS test need a real session (the auth gate).
- **Snap full flow unverified at runtime (chunk 07).** Upload → extract → realtime update, Arabic+English screenshot accuracy (PRD #4), PDF→PNG, and the 429 path all need a real session + real Snap screenshots + a live OpenAI call. `OPENAI_API_KEY` is set, the function is deployed, but nothing has been run end-to-end.
- **Private-bucket download by the edge function is unverified (chunk 07).** The chosen secure design has the edge function download the user's image from the private `snaps` bucket (as the user, via `edgeFunctionToken`) → base64 → OpenAI. If authenticated download from a private bucket doesn't behave as expected, extraction will mark the report `failed`; confirm on the first real run. (Fallback if it fails: a signed URL, or a public bucket with unguessable keys.)
- **Snap realtime REMOVED (chunk 07 review, migration `20260609210000`).** The per-user channel was a cross-tenant subscription gap and couldn't be RLS-gated (`realtime.channels` is owned by `postgres`; `project_admin` can't add a subscribe policy). Since the extract edge function is synchronous (the client awaits its result), realtime was redundant — dropped the trigger/function/channel; the UI now updates from the awaited invoke + `onSettled` refetch (request/response, not polling). Deviation from spec-E's realtime mechanism; can be re-added later via per-report unguessable-uuid channels if a fire-and-forget flow is wanted.
- **Rate-limit hardened: pending-only extraction (chunk 07 review).** The creation-count limit alone was bypassable by re-invoking on the same report id (repeat paid calls). The edge function now requires the report to be the caller's and `extraction_status='pending'` (else 409) before any OpenAI call, so each report is extracted at most once → total paid calls ≤ the hourly creation cap. Over the cap still marks the report `failed` + returns 429 (manual-entry path covers it).
- **OpenAI model pinned in the edge function (chunk 07).** `gpt-4o-mini` is hardcoded as a constant in `extract-snap-report`; revisit as models evolve / if extraction accuracy is poor.
- **`brandSchema` kept feature-local, not in `backend/shared` (chunk 02 deviation).** Spec 02-B says put the shared zod schema in `backend/shared`, but `zod` resolves only from `frontend/node_modules` (pnpm, not hoisted) and there is no Deno import map — a zod schema in `backend/shared` cannot be resolved by the frontend build or by Deno today, and brands has no edge function. Followed the chunk-01 precedent (`features/auth/auth.schema.ts`) and put it in `features/brands/brand.schema.ts`. When an edge function first needs to share an input schema (chunk 04/07), set up backend zod + a Deno import map and move shared schemas then.
- **Migrations live in `backend/migrations/` but the CLI looks in `./migrations` (cwd-relative).** Applying requires staging the `.sql` into root `./migrations`, running `db migrations up`, then cleaning up (root `./migrations` is an empty, untracked dir). Used this workaround again in chunk 03; still worth a permanent fix (a CLI migrations-dir setting or relocating the canonical dir) before more chunks pile on.

For a greenfield in 2026 the current stable majors are the right default and interoperate cleanly (build/typecheck/lint verified). If a specific dependency needs React 18 later, revisit then.

- **Edge runtime is Deno** → no server-side PDF rasterization. PDFs converted to PNG **client-side (pdf.js)**; edge function always gets an image.

- **Dates:** store ISO/Gregorian (UTC); display **Hijri (Umm al-Qura) + Gregorian**, Hijri leading in Arabic locale.
- **Reminders created in application/edge code, not Postgres triggers** (testable, visible).
- **Profile table named `app_users`** (auth `users` is InsForge's; `app_users` is our extension).
- **DB transactions on a BaaS** = Postgres functions (RPC) called from edge functions; canonical case is `mark-payment-received`.
- **Atomic mutation pattern** established for any multi-row all-or-nothing write.
- **Deploy:** frontend on Vercel; backend on InsForge. Accounts already provisioned (InsForge, OpenAI, Google OAuth).
- **RLS = standard Postgres `CREATE POLICY` (chunk 01).** Confirmed working on InsForge: per-operation policies `TO authenticated` with `user_id = (SELECT auth.uid())` in `USING`/`WITH CHECK`. Shipped in the same migration as `app_users` (`20260608172000_create-app-users.sql`), plus explicit `GRANT`s to `authenticated`. Verified live via `insforge db policies`.
- **Auth specifics (chunk 01).** Google OAuth uses InsForge's built-in provider (own client credentials added in the dashboard for branding); email verification method = `code` (6-digit OTP); password min length 6. `signInWithOAuth` redirects to `${origin}/dashboard`; `getCurrentUser` drives session hydration on load.
- **Dual-calendar dates (chunk 01).** `lib/date.ts` formats via `Intl.DateTimeFormat` with `islamic-umalqura` + `gregory`, UTC time zone; Hijri leads when locale is `ar`. `lib/currency.ts` formats SAR via `Intl.NumberFormat` (ar-SA / en-US).
- **Reusable CRUD pattern (chunk 02) — chunks 03–08 copy this.** Per table: migration with `id uuid default gen_random_uuid()`, `user_id` FK NOT NULL, 4 owner-only RLS policies (`TO authenticated`, `user_id = (SELECT auth.uid())`), a `(user_id)` index, and `GRANT`s — all in one migration. Frontend: `features/<domain>/` holds `<domain>.schema.ts` (zod: a form schema with all-string fields + a row schema that parses every API response), `<domain>.types.ts` (z.infer), `<domain>.api.ts` (insforge.database calls that filter by `user_id` AND rely on RLS — defense in depth, no IDOR; empty optionals mapped to NULL at the boundary). Data access is a `hooks/use<Domain>.ts` TanStack hook keyed from `constants/queryKeys.ts`; the view never calls the API directly. Forms use react-hook-form + zodResolver, localized errors, submit disabled while pending; lists have a designed empty state.
- **Add/edit UI = lightweight modal, no new dep (chunk 02).** `BrandFormDialog` is a self-built accessible overlay (`role="dialog"`, Escape-to-close, backdrop click), keyed by brand id so it remounts with fresh defaults — chosen over adding `@radix-ui/react-dialog`. shadcn currently only ships `Button`; revisit a real Dialog primitive if modals proliferate.
- **Status machine = one module, symmetric recompute (chunk 03).** All deal-status logic lives in `features/deals/status.ts`; status strings are never written ad-hoc. `computeDealStatus(current, deliverables)` is a pure function: `paid`/`cancelled` are sticky (never overridden — `paid` is owned by chunk 04, `cancelled` is manual), otherwise status is recomputed from deliverables on every change (≥1 posted → in_progress; all posted → posted; none → pending). The recompute is symmetric (un-posting moves status back down), which is the natural reading of "recompute on every deliverable change"; the spec only lists the forward transitions but doesn't forbid the reverse.
- **Deliverable = one checklist entry (chunk 03).** Each deliverable `{type, count, posted_at}` is a single posted/not-posted unit (`posted_at` is the signal); "2 stories" the acceptance test wants = two entries. The create form uses a react-hook-form field-array; `count` is a label/quantity, not per-unit posting.
- **Deals month filter targets `deadline` (chunk 03).** The `(user_id, status, deadline)` index backs server-side filtering; `month` ("YYYY-MM") becomes a half-open `deadline` range (`>= first day`, `< next month`). Deals with a null deadline are excluded when a month filter is active.
- **Brand detail screen added (chunk 03, deferred from 02).** Chunk 02 skipped the optional brand-detail screen; chunk 03-F needed it, so `/brands/:brandId` (`BrandDetailPage`) now lists the brand's deals + count (read-only); brand list names link to it. Lifetime SAR rollup still deferred to chunk 05.
- **Snap extraction pipeline (chunk 07).** Private `snaps` bucket; client validates by MIME + magic bytes + 10 MB cap and converts PDF→PNG (pdf.js, **dynamic-imported** so it's a lazy chunk, not in the main bundle); uploads to `userId/<uuid>`. The `extract-snap-report` edge function downloads the image **as the user** → base64 → OpenAI `gpt-4o-mini` with **structured outputs** (fixed JSON schema) + a bilingual, prompt-injection-safe system prompt; enforces a **per-user 20/hr** limit (count of `snap_reports` in the last hour) **plus a pending-only guard** (each report extracted once) → 429/409. UI updates come from the awaited synchronous invoke + `onSettled` refetch (realtime was removed in review — see Known issues). The user always has the final word — per-field manual override sets `extraction_status='manual'` (the editor reseeds from the latest report on open, so updates never overwrite newer values).
- **Reports = aggregate RPCs, same month basis as chunk 05 (chunk 08).** `report_monthly()` (12-month invoiced-vs-collected, zero-filled via `generate_series`) and `report_by_brand()` (brand×month deal count + invoiced + collected) are STABLE SECURITY INVOKER, scoped by `auth.uid()`; deals counted/invoiced by `created_at` month, collected by `received_date` month. Collection rate is derived in the client (`collected/invoiced`) with a divide-by-zero guard (invoiced 0 → "—"). Recharts is RTL-tolerant via `reversed` X-axis + right-oriented Y-axis when locale is `ar`; series colors are named constants (`REPORT_CHART_COLOR`).
- **Rollups = aggregate RPC, not client-side summing (chunk 05).** Top-line numbers come from the `dashboard_summary` SECURITY INVOKER + STABLE function (one round-trip; RLS-scoped), never by fetching all rows and summing in JS. "Needs attention" lists are single filtered queries (overdue payments) or derived from an already-fetched list (past-deadline deals) — never per-row queries (no N+1). Plain counts localize via `lib/currency.ts#formatNumber`.
- **Reminder creation = one helper, in code, never a trigger (chunk 06).** `features/reminders/reminder.api.ts#createReminder` is the single insert path; `createMeetingReminder`/`syncMeetingReminder`/`clearMeetingReminder`/`createPaymentReminder` wrap it. Meetings carry their reminder via `ref_table='meetings'` + `ref_id=meeting.id` (polymorphic, no FK). The dashboard "Today" reads not-done reminders due ≤ now+24h. Confirmed via `pg_trigger`: no triggers on `meetings`/`reminders`.
- **Cross-tenant FK lesson applied up front (chunk 06).** Following the chunk-04 payments review, `meetings` insert/update policies require any linked `brand_id`/`deal_id` to be owned by the caller (`brand_id IS NULL OR EXISTS(... owned)`), so this class of bug didn't ship in this chunk. (The same hardening for `ad_deals.brand_id`, from the chunk-04 review, is still an open follow-up.)
- **Atomic-write pattern (chunk 04) — later atomic writes copy this.** Multi-row all-or-nothing writes = one plpgsql function (RPC) defined in the table's migration, called from a thin edge function. The RPC is `SECURITY INVOKER` so `auth.uid()` + RLS scope it to the caller (plus explicit `user_id = auth.uid()` filters); the whole function body is one transaction, so a `RAISE` mid-way rolls everything back. Canonical case: `mark_payment_received` (set payment received → flip deal to `paid` when all received).
- **Edge function pattern (chunk 04) — the API we own.** Deno, `export default async (req) => Response`, imports via `npm:` specifiers (pinned, e.g. `npm:@insforge/sdk@1.3.1`). Deploy is **single-file** (`functions deploy <slug> --file …`), so the response envelope + HTTP codes are **inlined** (can't import `backend/shared` — docs say relative import, but the CLI only uploads one file; logged as a gap). Shape: CORS/OPTIONS → require `Authorization: Bearer` → `createClient({ edgeFunctionToken })` + `getCurrentUser` → zod-validate body → call RPC → return `{ok,data}`/`{ok,error}` with correct status (200/400/401/404/500). Client invokes via `insforge.functions.invoke(slug, { body })`.
